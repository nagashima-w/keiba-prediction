# 競馬期待値分析ツール 実装仕様書 v2

## 目的
netkeibaのデータをスクレイピングし、Claude APIで各馬の複勝圏内確率を推定、オッズとの乖離から期待値プラスの馬券を抽出するツール。**的中率ではなく回収率(期待値)を最大化する**設計。デスクトップUIで操作し、結果はDiscordにも通知できる。

## 設計思想
- JRAの控除率(約20〜25%)により、人気通りに買うと回収率は75〜80%に収束する
- 勝ち馬を当てるのではなく「市場(オッズ)が過小評価している馬」を見つける
- 数値データのスコアリングはコード内で決定論的に行い、LLMにはテキスト情報(調教評価・陣営コメント・展開想定等)の解釈と確率補正を任せるハイブリッド構成

## 技術スタック
- **言語: TypeScript(Node.js)で統一**
  - 理由: スクレイピング(cheerio + undici)、デスクトップUI(Electron)、Discord bot(discord.js)がすべて同一言語・同一コアロジックで書ける
- **コア**: `packages/core` — scraper / scorer / analyzer / ev をUIから独立したライブラリとして実装(monorepo構成、pnpm workspace)
- **UI**: Electron + React。コアを直接importして使う
- **LLM**: Anthropic API(claude-sonnet-4-6)
- **通知**: Discord Webhook(分析結果のプッシュ用)
- **データ保存**: SQLite(better-sqlite3)。スクレイピングキャッシュ・分析履歴・検証結果を保存
- **配布**: GitHub ActionsでWindows向けにelectron-builderでビルド
- **将来オプション**: Discord bot化はコアをそのままdiscord.jsで包む構成で対応可能。ただしCloudflare Workersはスクレイピングに不向き(CPU時間制限、リクエスト間隔制御の難しさ、データセンターIPのブロックリスク)なため、bot化する場合は自宅PC常駐または小型VPSを推奨。まずはElectron内蔵のWebhook通知で十分

## アーキテクチャ

### 1. scraper — netkeibaスクレイパー
- 入力: レースID(例: 202605020811)または開催日
- 取得対象:
  - 出馬表: 馬名・馬番・枠・騎手・斤量・馬体重・**各馬の個別ページ(db.netkeiba.com/horse/xxxx)へのリンク**
  - **各馬の全戦績(個別ページから取得)**: 着順・タイム・上がり3F・コース・距離・馬場状態・通過順位・レース間隔・開催月・馬体重・**枠番/馬番**。スコアリングの重み付けは直近を重視するが、ローテーションパターンや枠順適性の分類にはキャリア全体を使うため全走取得する(1頭1ページで取れるので追加リクエストは頭数分のみ)
  - 騎手・調教師の当該コース成績
  - 当日の天候・馬場状態(前日分析時は「想定馬場」として稍重以下の可能性をフラグ化)
  - 各馬の馬場状態別成績(良/稍重/重/不良)
  - 厩舎所在地(美浦/栗東)
  - **調教データ**: 調教タイム(コース・ラップ)、調教本数、併せ馬の結果、調教評価。※netkeibaでは一部プレミアム領域のため、無料で取得できる範囲を実装し、取れない項目はスキーマ上optionalにする
  - **陣営コメント・厩舎談話**: 取得可能なページから収集(こちらもoptional扱い)
  - 単勝・複勝オッズ(発走直前に再取得できる構造にする)
- スクレイピング起点: PC版の競馬新聞ページ `race.netkeiba.com/race/newspaper.html?race_id=...`(出馬表+各馬過去走+horse_idリンクが1ページに集約されており効率的)。調教は `?pid=oikiri`、厩舎コメントは `?pid=comment`、全戦績は `db.netkeiba.com/horse/{horse_id}`
- 制約(必須):
  - リクエスト間隔は最低1.5秒、User-Agent明示
  - HTMLパースは cheerio、セレクタは1ファイルに集約(構造変更対策)
  - **動的レンダリング対策**: netkeibaのレースページはJS(Riot.js)によるクライアントサイド描画部分があり、特にオッズは静的HTMLに含まれない可能性が高い。実装時はまず静的HTMLで取れる範囲を確認し、取れない要素は (a) ブラウザ開発者ツールで内部APIエンドポイント(XHR)を特定して直接叩く、(b) それが困難な場合のみPlaywrightでのヘッドレス取得にフォールバック、の順で対応する。Phase 1で要素ごとに「静的/API/Playwright」のどれで取るかを確定させること
  - 取得結果はSQLiteにキャッシュ(同一レースの再取得を避ける)

### 2. scorer — 数値スコアリング(決定論的に計算)
基礎スコア:
- 近走着順(重み減衰付き: 直近ほど重い)
- 上がり3F順位
- コース・距離適性(同条件での複勝率)
- 騎手の当該コース複勝率
- 斤量変化、馬体重増減
- 枠順バイアス(コースごとの内外有利データを定数テーブルで保持)

環境・状態バイアス補正:

**馬場状態適性**
- 当日(または想定)馬場が稍重以下の場合、各馬の道悪実績で補正
- 道悪実績2走未満は「不明」として補正なし(サンプル不足でのペナルティは付けない)
- 芝とダートで別集計。馬場悪化はダートでは時計が速くなる方向に働く点に注意

**競馬場適性**
- 当該競馬場での複勝率(出走2回以上ある場合のみ)
- 出走歴がない場合はコース特性の類似性で代替評価: 回り方向(右/左)、直線の長さ、坂の有無(中山・阪神・中京は急坂)、洋芝(札幌・函館)/野芝
- 競馬場ごとのコース特性は定数テーブルで保持

**輸送・滞在バイアス**
- 厩舎所在地×開催場から輸送負荷を分類: 地元圏 / 短距離輸送 / 長距離輸送
- 長距離輸送を伴った過去走の複勝率で補正
- 滞在競馬(札幌・函館開催): 滞在開催での過去実績があれば使用。輸送に弱い馬は滞在競馬でプラス補正
- 輸送を伴うレースで-10kg以上の馬体重減が複数回あれば「輸送弱」フラグ

**季節適性**
- 過去走を開催月から季節分類: 夏(6〜9月) / 冬(12〜2月) / 春秋(その他)
- 各季節での複勝率を集計し、今回の開催季節に対応する成績で補正
- 夏場に馬体重が減りやすい馬(夏開催で平均-6kg以上)は「夏負け」フラグでマイナス補正
- 各季節2走未満は補正なし

**ローテーション適性(鉄砲/叩き良化/使い込み下降)**
- 全戦績からレース間隔を分類: 連闘〜中3週 / 中4〜9週 / 休み明け(中10週以上)
- 休み明けを起点に「休み明け何走目か」を各過去走にラベル付けし、走目ごとの複勝率カーブを作る
- カーブの形状からタイプを分類:
  - **鉄砲型**: 休み明けの成績が通常時と同等以上
  - **叩き良化型**: 休み明けは凡走し、2〜3走目でピーク
  - **使い込み下降型**: 使い続けると成績が落ちる(4走目以降の複勝率がピーク時より明確に低い)。鉄砲型と重複しうる(休み明けがピークでそこから下降)
- 今回が休み明け何走目かに応じて補正:
  - 鉄砲型 × 休み明け → 補正なし(オッズが人気を落としていれば期待値的に妙味)
  - 叩き良化型 × 休み明け → マイナス補正
  - 叩き良化型 × 叩き2〜3走目 → プラス補正
  - 使い込み下降型 × ピーク走目を過ぎた出走(目安: 休み明け4走目以降) → マイナス補正
- 休み明け実績2走未満は「不明」として弱いマイナス補正のみ(休み明けは平均的に割引が妥当なため)

**枠順適性(馬個別)**
- コースレベルの枠順バイアス(基礎スコア側)とは別に、馬個別の内外適性を全戦績から集計
- 過去走の枠番を内枠(1〜3枠)/中枠(4〜6枠)/外枠(7〜8枠)に分類し、それぞれの複勝率を算出
- 今回の枠に対応する成績で補正(例: 外枠複勝率が明確に低い馬が今回8枠 → マイナス補正)
- 各ゾーン2走未満は補正なし
- 枠の影響は2層で独立に評価する: ①コースレベルの枠順バイアス(基礎スコア側の定数テーブル。コース形態によって全馬に効く)、②本項の馬個別の枠別成績。両者は別々の補正項として加算し、重みも個別にconfigで調整可能とする

基礎スコア+バイアス補正から各馬の複勝圏内確率の事前推定値(prior)を算出。各バイアスの重みはconfigで調整可能にし、verifyの結果を見ながらチューニングできる設計とする。

### 3. analyzer — Claude API補正
- scorerのpriorと、以下のテキスト情報をプロンプトに含める:
  - **調教内容**: 時計の速さ・ラップの質・併せ馬で先着したか・調教評価(「動き抜群」「重目残り」等の定性評価の解釈はLLMが得意な領域)
  - **陣営コメント・厩舎談話**: 強気/弱気のトーン、「今回は叩き台」「仕上がり途上」等のシグナル読み取り
  - レース間隔、脚質と展開想定(逃げ馬の数からペースを推定)
  - 当日の天候・馬場状態と、雨予報時の馬場悪化シナリオ
- LLMへの指示: 「各馬の複勝圏内確率をJSONのみで出力。priorから±10%以内の補正に留める。補正には必ず根拠(調教・コメント・展開)を明記」
- 出力スキーマ: `{"horses": [{"number": 1, "place_prob": 0.42, "reason": "..."}]}`
- JSONパース失敗時は1回リトライ、失敗時はpriorをそのまま使用(フェイルセーフ)
- 1レース1リクエストにまとめてコスト管理

### 4. ev — 期待値計算
- 複勝期待値 = place_prob × 複勝オッズ(下限値を使用)
- EV > 1.0(設定可能な閾値)の馬券のみ抽出
- 全推定結果をSQLiteに保存し、レース後に実際の結果と突き合わせて回収率を記録するverify機能を用意

### 5. ui — Electronデスクトップアプリ
- 画面構成:
  - **レース選択画面**: 日付を選ぶと当日の開催・レース一覧を表示
  - **分析画面**: レースを選んで「分析実行」→ 進捗表示(スクレイピング→スコアリング→LLM分析)→ 結果テーブル(馬番・馬名・prior・補正後確率・オッズ・EV・LLMの根拠)。EVプラスの行をハイライト
  - **検証画面**: 過去の分析履歴と実際の結果、累積回収率のグラフ、推定確率帯ごとの実際の複勝率(キャリブレーション表)
  - **設定画面**: APIキー、Discord Webhook URL、EV閾値、各バイアスの重み
- 「Discordに送信」ボタンで分析結果をWebhook送信(自動送信のON/OFF設定も)

## フェーズ分割(この順で実装)
1. **Phase 1**: core/scraper + JSONダンプのCLI(データが正しく取れることを最優先で確認)
2. **Phase 2**: core/scorer + ev(LLMなしで期待値抽出が動く状態)
3. **Phase 3**: core/analyzer(Claude API組み込み、調教・コメント解釈)
4. **Phase 4**: Electron UI(分析画面→検証画面→設定画面の順)
5. **Phase 5**: Discord Webhook通知 + GitHub Actionsビルド
6. **Phase 6(オプション)**: discord.js botフロントエンド(`/analyze` スラッシュコマンド)

## 注意事項
- netkeibaのスクレイピングは規約上グレーのため個人利用専用。README等に明記
- 馬券購入の自動化は行わない(分析・通知まで)
- verify結果に「推定確率帯ごとの実際の複勝率」を出力すること(例: 推定40〜50%の馬が実際に42%複勝圏に入っていれば健全)
- バイアス項目が多いため過剰補正に注意。各バイアスの寄与度をログ出力し、verifyで重みを調整できること
