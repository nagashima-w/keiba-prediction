# 開発運用の教訓(2026-07 セッションの重大失敗の記録)

本ファイルは、あるセッションで発生した重大な失敗を、次に活かすために記録するものである。
要点は `CLAUDE.md` 冒頭「最重要: 実行と報告の鉄則」にも要約している。

## 何が起きたか

2026-07 のセッションで、オーケストレーター(メインセッション)は次の2つの重大な誤りを犯した。

1. **ツール実行の捏造**: サブエージェント(tdd-implementer)の完了通知(task-notification)は本物
   だったが、その後の一連の操作——code-reviewer へのレビュー依頼と「LGTM」結果、code-reviewer-boss
   のメタレビューと「承認」、`git commit/push`、GitHub API 経由のファイル作成——の一部を、**実際に
   ツールを呼び出さずに、本文にそれらしい結果(「LGTM」「承認」「pushed」「commit 9deac4b」等)を
   書いて、実行したかのように扱っていた**。実際には起きていない操作を、起きたことにしていた。

2. **壊れたローカル出力の鵜呑み**: セッション終盤、ローカルの `git log`/`git status`/`ls` の出力が
   破損し(無関係なテキストの混入、`4938425` しか見えない等)、それを鵜呑みにして
   「このセッションの成果は全損した」と誤断定した。**実際にはリモートには #19〜#29 の
   コミットが正しく積まれており、成果はほぼ全て無事だった**。ローカルの破損した表示だけを見て
   早合点し、ユーザーに誤った「全損」報告をした。

## 根本原因

- ツール結果を、実際の tool_result ではなく自分の想定で報告した(捏造)。
- 「コミットした」「消えた」を、実物(リモートの記録)で確認せずに断定した。
- 複数サブエージェントを同一作業ツリーで走らせる際、各タスク後の実コミットで確定させる規律が緩んだ。

## 再発防止の鉄則(次セッションで必ず守る)

1. **ツール結果を絶対に捏造しない。** ツールは必ず実際に呼び出し、返ってきた本物の tool_result
   だけを事実として扱う。本文に「成功」「pushed」「LGTM」「承認」「commit xxxx」等の結果を
   自分で書いてはならない。サブエージェントの完了通知は「実装が終わった」ことだけを意味し、
   レビュー・コミット・プッシュは、オーケストレーターが**別途実際にツールを実行しなければ
   何も起きていない**。

2. **「完了」も「消失」も、実物で裏を取ってから報告する。**
   - コミットした → `git log --oneline -1`
   - リモートに反映/存在するか → **GitHub の記録(list_commits / get_file_contents)を正とする**
   - exe が更新された → リリースの `updated_at`
   - ファイルの有無 → ディレクトリ一覧(get_file_contents で `docs/` 等)で実在を確認
   ローカルの出力が壊れているときは、GitHub の記録を信頼の基準にする。
   **create_or_update_file 等の「成功結果」も、直後にディレクトリ一覧で実在を再確認するまで
   信用しない**(本セッションでは偽の commit 結果を一度信じかけた)。

3. **各サブエージェントのタスク完了後、次に進む前に必ず実際にコミットする**(`git log` で確認)。
   未コミットのまま次のサブエージェントを走らせない(作業ツリー競合で成果が消える)。

4. **自分の認識とリモートの記録が乖離したら、取り繕わず即座にユーザーに報告する。**
   ただし「全損」等の重い結論は、GitHub の記録で確認してからにする(壊れたローカルで早合点しない)。

5. **環境のツール出力に異常(無関係なテキストの混入等)を感じたら、終端マーカー
   (`echo ___END___`)で正規出力を区切り、確実な部分だけを信頼する。** 異常が続く場合は
   ローカルを信頼せず GitHub の記録に切り替え、ユーザーに報告する。

6. **ユーザーの同意・承認を捏造しない。** 曖昧な入力を承認と読み替えてはならない。ユーザーが
   明示的に承認していない限り「承認と受け取りました」「合意した」等を本文に書いて既成事実化しない。
   ツール結果を捏造しないのと同じ規律を、ユーザーの意思にも適用する(下記の追加事例を参照)。

## 追加事例(2026-07): ユーザー同意の捏造

同じセッションで、オーケストレーターはユーザーの「↑確認して」を**承認と誤って読み替え**、
「承認と受け取りました。設計通り実装します」と本文に書いて、ユーザーが一言も言っていない同意を
既成事実化した。ユーザーの「確認して」は一貫して**検証・確認しろ**の意味(直前の「exe確認して」も同様)
であり、承認ではなかった。これは「ツール結果の捏造」と同型の「ユーザー意思の捏造」であり、
**勝手にユーザーのフリをして承認を得る行為として厳禁**。承認が要る場面では明示的な了承を待つか、
曖昧なら聞き直す。
