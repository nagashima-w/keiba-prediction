### #30 結果取込: 未確定レースの区別(TDD)
- `#All_Result_Table` 要素はあるが着順行が0件のレース(発走前・確定前)を、構造異常
  (`RaceResultParseError`)と区別し `RaceResultNotConfirmedError` で扱う。UI は赤エラーでなく
  「まだ結果が確定していません(発走前・確定前)。確定後に再取込を」と穏やかに案内。着順は保存しない
  (取込済みにしない)。テーブル要素自体が無い場合は従来どおり構造異常。
- 未確定判定を IPC 越しに例外でなく `status: "not_confirmed"` の正常応答で返すと #31 と相性が良い。
- フィクスチャ: 発走前の地方 result.html(All_Result_Table ありで tbody 空)を1つ取得して固定する。

### #31 結果の一括取込(TDD)
- 分析済みで結果未取込のレースを列挙し直列取込。**未取込判定は「race_results に行が1件も無い」を

  のみのレースを未取込と誤判定する。絶対に使わない)。未確定(#30)は自動スキップ、部分失敗継続、
  1.5秒レート制限、境界でキャンセル可、取込後に検証レポート再読込。検証画面に「未取込をまとめて取り込む」

### #34 レース単位の予実ブレークダウン(TDD)

## 2. プロンプト改善の運用目安(ユーザーに説明済み・記録)
- **回収率より先にキャリブレーション(確率の当たり具合)を見る**。回収率は配当ばらつきで暴れるが、
  3着内率(二項)は素直に収束する。まず100〜200レースでキャリブレーションを確認し、確率推定が
  合ってきたら回収率(300点以上の賭け)で妙味を見る。
- プロンプトを実際にいじる判断は **1版あたり最低200〜300レース**(EVプラス賭け点300前後)。
  A/B比較で優劣を言うには各版300〜500点。少数データで回収率が良く見えても偶然を疑う。
- 中央だけだと週末中心で貯まりにくい。**地方(毎日開催)を分析対象に入れると早く貯まる**。ただし地方は
  条件が違う(#32で中央/地方を分けて見られるようにする狙い)。**偏った時期だけのデータで判断しない**。
- 検証画面の集計件数(includedAnalysisCount)・賭け点数(betCount)を、判断に足るデータ量の目安に使う。

## 4. 技術メモ(横展開の観点)
- 既存の nullable/特殊値の契約(finish_position は中止・除外で NULL、オッズは取消で非数値→null)を、
  新規コード(SQL集約・パーサー)が踏まえているか常に確認する。「値の有無」と「行の有無」の混同、
  SQLの `COUNT(列)` の NULL セマンティクスに注意(#31のバグ原因)。
- レース別表示の緑ハイライトは `isPositive` 基準(`ev !== null` では印だけの非EVプラス馬を誤表示)。
- 予想印の対抗「〇」は U+3007。LLM が `○`(U+25CB)/`◯`(U+25EF)を出す場合に備え正規化する。
- undici 8 は Node≥22.19 必須で Electron 34(Node 20)と非互換 → net.fetch アダプタ注入+undici ^7 整合で対応済み(既存コミットにあり)。
