# 検討: 検証結果(回収率)に基づくLLMプロンプト改善の仕組み(Task #24)

作成: 2026-07-14。ユーザー要望「検証した結果を見て回収率がよくない場合にLLMに投げるプロンプトを改善できるような仕組み」に対する設計検討。**本書は検討メモであり、実装着手前にユーザーと方式・範囲を合意する**。

## 大前提: サンプル数と過学習リスク(最重要)

- 現状 verify の実データはほぼ蓄積されていない(実運用これから)。**少数レースの結果を見てプロンプトをいじると確実に過学習する**(直近の当たり外れにプロンプトが引っ張られ、汎化性能はむしろ落ちる)。
- この道具の思想は「市場から独立した確率推定 × 市場オッズ」で妙味を取ること。プロンプト改善が「市場(人気)に寄せる」方向に働くと妙味そのものが消える(Task#22でアンカリング禁止を入れたのと同じ懸念)。
- したがって設計方針は **「まず記録と可視化を整え、プロンプトの実改変は判断に足るデータ量が貯まってから」** とする。記録・可視化(下記A・B)を先に入れ、実改変(C・D)は後追いにする順序を推奨。

## 現状でできること / 足りないもの

| 要素 | 現状 |
|---|---|
| キャリブレーション表(補正後確率帯ごとの実複勝率) | **既にverifyが出力**(`CalibrationBin`)。過信/過小評価の帯が見える |
| 補正の裏目分析の材料(prior・adjustedProb・mark・実着順) | **保存済み**(`StoredAnalysisHorse` + `race_results`)。集計軸が未実装なだけ |
| 回収率(実配当優先・推定EV除外) | 既にverifyが出力 |
| どのプロンプト版・モデルで分析したかの記録 | **なし**(分析レコードに版・モデル列がない) |
| プロンプトへの追加指示の注入口 | **なし**(`BuildPromptInput` に自由記述欄がない。プロンプトは固定構造) |
| 版ごと・条件ごとのverify比較 | なし(全分析を一括集計するのみ) |

## 方式候補

### A. プロンプト版・モデルの記録(土台・最優先)
- プロンプトテンプレートに版番号(例 `PROMPT_VERSION = "2026-07-14.1"`)を持たせ、`analyses` テーブルに `prompt_version` / `model` 列を追加(mark/ev_estimated と同じ冪等マイグレーション)。
- verify を `prompt_version` 別・`model` 別に集計できるようにする(「版ごとの回収率・キャリブレーション」比較)。
- **効果**: 「プロンプトを変えた→本当に良くなったのか」を数字で判定できる土台。これが無いと改善サイクルが機能しない。
- **コスト**: 中(DB列2つ+verify集計軸+テンプレ版定数)。

### B. 補正の誤り傾向の可視化(verify拡張・優先)
既存データだけで出せる、プロンプト改善の材料となる集計軸を verify に追加する:
- **補正方向×結果のクロス**: AIが確率を上げた馬(adjustedProb>prior)の実複勝率 vs 下げた馬(<prior)の実複勝率。「上げが裏目/下げが裏目」の傾向を数値化。
- **キャリブレーションの過信バイアス**: 既存のキャリブレーション表から「予測が実績を上回っている帯(過信)」を明示ラベル化。
- **印別的中率**: ◎の複勝率・勝率、☆注(穴)の的中率。印付けが機能しているかの指標。
- **効果**: 「どこをどう直すべきか」の材料を人間(と後述Dのループ)に提示。プロンプト改善の入力になる。
- **コスト**: 中(verify集計拡張+UI表示)。既存のadjustedProb/prior/mark/着順で完結し、新規スクレイピング不要。

### C. 追加指示の注入口(手動改善サイクル)
- `BuildPromptInput` に `additionalInstruction?: string`(ユーザー自由記述)を追加し、プロンプト末尾の指示ブロックに差し込む。設定画面から編集可能にする。
- Bで見えた傾向を元に「人気薄の複勝率を過大評価しがちなので、穴馬の3着内率は慎重に見積もること」等を、**コード変更なしで**注入。
- **効果**: 反復サイクルを高速化。ただし注入内容を変えるたびに実質「別プロンプト」になるため、**必ずA(版記録)と併用**し、注入文自体も版に含めて記録する(でないと何が効いたか追えなくなる)。
- **コスト**: 小〜中(注入口+設定永続化+プロンプト差し込み)。
- **注意**: 自由記述をそのままLLMに渡すため、アンカリング禁止など既存の設計思想と矛盾する指示を書けてしまう。設定画面に「市場オッズに確率推定を寄せない」旨の注意書きを添える。

### D. LLMによる改善提案(発展・任意)
- B の傾向サマリ + キャリブレーション表をテキスト化してLLMに渡し、「このプロンプトのどこをどう直すべきか」を提案させる半自動ループ。ユーザーが採否を判断して C の注入欄に反映。
- **効果**: 改善案の生成を自動化。
- **リスク**: 過学習(少数レースに引っ張られた提案)、市場迎合方向への漂流。**十分なデータ量が前提**で、採否は必ず人間が握る。
- **コスト**: 中〜大(傾向サマリ整形+提案用プロンプト+UI)。優先度は最後。

## 着手順序(2026-07-14 ユーザー合意)

**B → A → C** の順で実装する。D は将来やる意思あり(下記の設計上の含意に反映)。

1. **B(補正の誤り傾向の可視化)を最初に実装**。既存データ(prior・adjustedProb・mark・着順)だけで完結し、DB変更なしで価値が出せるため先行する。→ Task #26
2. **A(プロンプト版の記録 + verify版別比較)**。ただし比較軸は **prompt_version 別のみ**(model 別比較は入れない=合意)。→ Task #27
3. **C(追加指示の注入口)**。手動改善サイクル。必ず A の版記録と併用し、注入文も版に含める。→ Task #28
4. **D(LLM自動提案)は将来**。やる意思があるため、**B の傾向サマリは最初から機械可読な構造(`VerifyTrendReport` 等の型)で設計**し、後でそのままLLMへ渡せる形にしておく(Task #26の設計制約)。

データが判断に足る量(最低でも数十レース、できれば100レース超)貯まるまで、C での実プロンプト改変は慎重に。B・A の観測基盤を先に、が結論。

## 合意結果(2026-07-14)
1. 着手範囲・順序: **B → A → C**(上記)。
2. verify比較の粒度: **prompt_version 別のみ**(model 別は不要)。
3. D(LLM自動提案): **将来やる** → B の傾向サマリを機械可読な構造で設計しておく。
