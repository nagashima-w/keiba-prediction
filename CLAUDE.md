# プロジェクトルール: 競馬期待値分析ツール

仕様は `keiba-ev-tool-spec.md` に従うこと。実装はフェーズ順(Phase 1→6)に進める。

## 言語規約
- コードコメント、ドキュメント、コミットメッセージ、テスト名(describe/it)はすべて日本語で書く
- 変数名・関数名は英語(通常のTypeScript慣習)で良い

## 開発手続き: テスト駆動開発(必須)
すべての機能実装は以下の手順を厳守する。手順を飛ばしてはならない。

1. **Red**: 仕様から期待される振る舞いをテストとして先に書き、テストが失敗することを実行して確認する
2. **Green**: テストを通す最小限の実装を書き、テストが通ることを実行して確認する
3. **Refactor**: テストが通る状態を維持したままリファクタリングする
4. テスト実行結果(失敗→成功)を必ずログとして報告する

- テストフレームワークは vitest を使用
- scraperのテストは保存済みHTMLフィクスチャ(`fixtures/`)に対して行い、実サイトへのリクエストはテストに含めない
- scorerの各バイアス計算は、境界値(サンプル2走未満で補正なし等)を含むテーブル駆動テストを書く

## レビュー体制(必須)
- **実装を行ったコンテキストと同じコンテキストでレビューを完結させてはならない**
- 各機能の実装(Red→Green→Refactor)が完了するたびに、`code-reviewer` サブエージェントを起動して批判的レビューを受ける
- レビュー指摘のうち「重大」「要修正」は `tdd-implementer` が修正し、修正後に再度 `code-reviewer` のレビューを受ける。「提案」は対応可否を判断して記録する
- レビューが承認(LGTM)になるまで次の機能に進まない

## ワークフロー(オーケストレーション)
メインセッションは進行管理に徹し、実装とレビューはサブエージェントに委譲する:

1. 仕様書から次に実装すべき機能を1つ選び、受け入れ条件を整理する
2. `tdd-implementer` サブエージェントに実装を委譲(TDD手順の遵守を指示に含める)
3. 完了報告を受けたら `code-reviewer` サブエージェントにレビューを委譲
4. 指摘があれば 2 に戻る。LGTMなら日本語でコミットして次の機能へ
5. 各Phase完了時に、Phase全体の統合動作確認を行ってからユーザーに報告する

## その他
- netkeibaへの実リクエストを伴う動作確認は、ユーザーに確認してから最小限の回数で行う
- 依存パッケージの追加時は理由を一言記録する
